Apps

Das letzte Buchkapitel: aus einer Spiele-Demo eine installierbare App machen. Original: A3GPU215_Apps (MarbleMaze, iSlidingPuzzle — beides Adobe-AIR-Pakete mit App-Icons und Manifest-Datei).

Zwei Projekte, eine Lektion

iSlidingPuzzle ist inhaltlich dasselbe Spiel wie Chapter 06 · Sliding Puzzle — hier ging es dem Buch nicht um neue Spiellogik, sondern ums Verpacken: App-Icons in mehreren Größen, eine -app.xml-Manifestdatei, Touch statt Maus. MarbleMaze ist das einzige inhaltlich neue Spiel — und das einzige, das wirklich einen Sensor braucht, den es auf dem Desktop nicht gibt. Deshalb liegt der Fokus dieser Kapitelseite auf Marble Maze plus einer Erklärung, wie sich das AIR-Verpackungsprinzip auf eine Web-App überträgt.

Sensor mit Fallback: dasselbe Muster wie im Original

MarbleMaze.as prüft Accelerometer.isSupported und benutzt auf dem Desktop stattdessen die Pfeiltasten — Adobe AIR lief schließlich auch auf Windows/Mac-Rechnern ohne Sensor. Genau dasselbe Muster gibt es im Web: die DeviceOrientation-API auf Mobilgeräten (auf iOS erst nach expliziter Nutzer-Bestätigung per Tap), Pfeiltasten als Fallback überall sonst.

ActionScript 3 / AIR 
if (Accelerometer.isSupported) {
  accelerometer = new Accelerometer();
  accelerometer.addEventListener(
    AccelerometerEvent.UPDATE,
    accelerometerHandler);
} else {
  stage.addEventListener(
    KeyboardEvent.KEY_DOWN,
    keyDownFunction);
}
JavaScript / Web 
if (window.DeviceOrientationEvent
    && matchMedia(
      "(pointer:coarse)").matches) {
  window.addEventListener(
    "deviceorientation",
    e => this.onTilt(e));
} else {
  window.addEventListener(
    "keydown",
    e => this.keyDown(e));
}

Ein Unterschied bleibt: iOS verlangt seit iOS 13 eine explizite Nutzerfreigabe für Bewegungssensoren (DeviceOrientationEvent.requestPermission()), auslösbar nur durch eine echte Nutzeraktion wie einen Tap — deshalb der "Neigen aktivieren"-Button unten, den es im Original nicht brauchte.

AIR-Verpackung → PWA-Manifest

MarbleMaze-app.xml beschrieb App-Name, Version, Icons und Berechtigungen für Adobe AIR. Die direkte Web-Entsprechung ist eine manifest.json — dasselbe Grundprinzip (Metadaten + Icon-Set, die den Browser "installierbar machen"), nur JSON statt XML: 

{
  "name": "Marble Maze",
  "short_name": "MarbleMaze",
  "start_url": "./index.html",
  "display": "standalone",
  "background_color": "#141a22",
  "theme_color": "#5fe0c9",
  "icons": [
    { "src": "icons/icon-192.png", "sizes": "192x192", "type": "image/png" },
    { "src": "icons/icon-512.png", "sizes": "512x512", "type": "image/png" }
  ]
}

Mit dieser Datei (verlinkt per <link rel="manifest">) und einer HTTPS-Auslieferung — beides bei GitHub Pages automatisch gegeben — bietet der Browser "Zum Startbildschirm hinzufügen" an, genau wie ein AIR-Installer eine native App installierte. Für echte Offline-Fähigkeit käme zusätzlich ein Service Worker dazu — eine eigene, größere Technik, die den Rahmen dieses Buch-Ports sprengt, aber der letzte logische Schritt wäre.
