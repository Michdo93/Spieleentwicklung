Dungeon 3D

Ein waschechter First-Person-Walker: frei drehen, vor/zurück laufen, Münzen einsammeln, Hindernissen ausweichen. Original: A3GPU214_Dungeon3D.

Von Flashs 3D-Engine zur Kamera-Matrix von Hand

Das Original bewegt nicht den Spieler, sondern die Welt in die Gegenrichtung (worldSprite.x = -newX) und dreht die Kamera über viewSprite.rotationY — Flash übernimmt den Rest automatisch. Ohne diese Engine rechnet project() jeden Weltpunkt explizit in Kamera-Koordinaten um (Verschiebung relativ zur Kameraposition, dann Rotation um den Blickwinkel), bevor die Formel aus Chapter 14a die eigentliche Perspektive anwendet.

"Squares" sind Hindernisse, keine Räume

Ein Detail, das beim ersten Lesen des Originals leicht übersehen wird: Square-Objekte sind keine begehbaren Zimmer, sondern solide Blöcke — für jeden wird per addWall() an allen vier Kanten eine Wand erzeugt. Der Spieler bewegt sich frei im offenen Raum zwischen diesen Blöcken. Das macht Dungeon 3D strukturell näher an Top-Down Game aus Kapitel 12 (dieselbe Rechteck-Kollision!) als an ein klassisches Wand-Labyrinth.

ActionScript 3 
if (object is Square) {
  addWall(object.x+object.width/2,
    object.y, object.width, 0);
  addWall(object.x,
    object.y+object.height/2,
    object.height, 90);
  // ... zwei weitere Kanten
  squares.push(object);
}
JavaScript 
blocks.forEach(b => {
  quads.push(wallQuad(
    b.x, b.z, b.x+b.w, b.z));
  quads.push(wallQuad(
    b.x+b.w, b.z,
    b.x+b.w, b.z+b.d));
  // ... zwei weitere Kanten
});

Tiefensortierung: Painter's Algorithm statt Matrix-Abfrage

zSort() im Original fragt für jedes Objekt die tatsächliche 3D-Tiefe über getRelativeMatrix3D() ab und sortiert danach, damit nahe Objekte richtig vor fernen gezeichnet werden. Hier reicht ein einfacher Vergleich der berechneten Kamera-Tiefe (depth) — dasselbe Ziel, ohne Matrix-API.
